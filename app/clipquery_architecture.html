ClipQuery — Full Product Model

Two-tier commercial architecture · Built on current v1 codebase · Reference for future implementation

The Two-Tier Model

Tier 1

BYOK — Pipeline Access

You are a pure compute layer. User brings all their own API keys (Groq, HuggingFace, Pinecone). You run the pipeline, push vectors to their Pinecone, return confirmation. Zero data ownership on your side. Billed per second of video processed. You only store: their API key (hashed) + usage seconds.

Tier 2

Premium — Managed Platform

You are the platform. User logs in, uploads video, searches your shared index. You own the Pinecone DB, S3, all API keys. User-scoped data isolation via metadata filters. Billed via subscription. Full YouTube-like UX: upload, search, browse public library.

Key insight: Both tiers share the same core pipeline code (your existing services/ modules). What changes is only the key source (user's vs yours), the storage destination (their Pinecone vs yours), and the auth mechanism (API key vs JWT session).

Database Schema

Postgres for structured data · Pinecone for vectors · S3 for files

BYOK Tables

-- Who has access to the pipeline CREATE TABLE byok_api_keys ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), api_key_hash TEXT NOT NULL UNIQUE, -- store hashed, never raw owner_email TEXT NOT NULL, plan TEXT DEFAULT 'free', -- free | starter | pro credit_seconds INT DEFAULT 0, -- prepaid seconds remaining created_at TIMESTAMPTZ DEFAULT now(), is_active BOOLEAN DEFAULT true ); -- Per-request usage audit log CREATE TABLE byok_usage_log ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), api_key_id UUID REFERENCES byok_api_keys(id), video_name TEXT, duration_sec INT NOT NULL, -- extracted via ffprobe BEFORE pipeline processed_at TIMESTAMPTZ DEFAULT now(), status TEXT DEFAULT 'success' -- success | failed | refunded );

Premium Tables

-- Platform users (sign up via email/OAuth) CREATE TABLE users ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), email TEXT NOT NULL UNIQUE, password_hash TEXT, -- null if OAuth login oauth_provider TEXT, -- google | github | null plan TEXT DEFAULT 'free', -- free | premium created_at TIMESTAMPTZ DEFAULT now(), is_active BOOLEAN DEFAULT true ); -- Uploaded video metadata (Pinecone holds vectors, this holds metadata) CREATE TABLE videos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID REFERENCES users(id), title TEXT NOT NULL, s3_key TEXT NOT NULL, -- path in your S3 bucket duration_sec INT NOT NULL, status TEXT DEFAULT 'processing', -- processing | ready | failed is_public BOOLEAN DEFAULT false, -- for public library browsing pinecone_namespace TEXT, -- namespace in master index uploaded_at TIMESTAMPTZ DEFAULT now() );
Critical: BYOK requests are stateless — you never store the user's Groq/HuggingFace/Pinecone keys in your DB. They come in on the request body and are discarded after the pipeline completes. You only store the audit log above for billing.

BYOK Upload Request — Exact Shape

This is what a developer's product sends to your API:

Headers: X-API-Key: cq_live_xxxxxxxxxxxxxxxxxxxx ← your issued key Content-Type: multipart/form-data Form Body: video_file: <binary> groq_api_key: gsk_xxxxxxxxxxxxxxxxxxxx hf_token: hf_xxxxxxxxxxxxxxxxxxxx pinecone_api_key: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx pinecone_index: their-index-name namespace: optional-namespace Your pipeline does: 1. ffprobe → extract duration_sec 2. Check credit_seconds >= duration_sec ← gate here, fail fast 3. Run pipeline with their keys 4. Push vectors to THEIR Pinecone 5. Deduct duration_sec from credit_seconds 6. Insert row into byok_usage_log 7. Return { job_id, duration_sec, credits_remaining }

What Changes in Your Existing Codebase

File / Module Current State What to Add for BYOK
core/config.py Loads master API keys from env No change — master keys stay here. BYOK keys come from request, not config.
core/auth.py Single APIKeyHeader dependency Add a second dependency: get_jwt_user() for premium. Router decides which to use per endpoint.
services/pinecone_integrate/ Hardcoded to master Pinecone keys Refactor pincone_initialize.py to accept optional key override. BYOK passes user key; Premium passes None (uses config).
services/orchestrate_pipeline_db.py Uses config keys implicitly Accept a KeyContext dataclass (groq_key, hf_token, pinecone_key, index_name). Thread it through the pipeline instead of reading from config directly.
api/v1/endpoints/ingest.py Single upload endpoint, Premium only Add tier detection: if request has BYOK key → extract user keys from form → build KeyContext → pass to pipeline.
schemas/ Premium-only schemas Add BYOKUploadRequest (includes key fields). Add BYOKUsageResponse.
main.py Single router Mount a second router prefix: /byok for BYOK-only endpoints alongside existing /api/v1.
The pipeline itself doesn't change. Your processor_raw.py, processor_final_data.py, all audio/visual services — they remain identical. You're only changing which keys get injected into them. This is clean separation of concerns.

Pinecone Strategy — BYOK vs Premium

Aspect BYOK Premium
Index Owner User owns it — their account, their billing You own it — master index on your account
Isolation Complete — different Pinecone account entirely Namespace per user_id within your master index
You pay for it? No Yes — factor into Premium pricing
Multi-tenant filter Not needed — user queries their own index directly Your existing $or filter (user_id + public_listing) — already built
Future consideration: For Premium, use Pinecone namespaces (namespace=user_id) instead of metadata filters for isolation. Namespaces are index-level partitions — faster and cheaper than metadata-filtered scans at scale. Your current $or filter works fine for now, migrate later when user count grows.

Billing Model

BYOK Billing

Prepaid Credits

User buys X minutes upfront via Stripe. Pipeline runs only if credit balance ≥ video duration. No credit → 402 Payment Required, pipeline never starts.

Suggested pricing:
₹5 / minute processed
Starter: 60 min = ₹300
Pro: 600 min = ₹2500

Premium Billing

Monthly Subscription

Monthly flat fee via Stripe Subscriptions. Free tier: 10 videos/month. Premium: unlimited. Platform manages all costs internally.

Suggested pricing:
Free: 10 videos/month
Premium: ₹199/month

Stripe integration point: One webhook handler endpoint (POST /webhooks/stripe) handles both: credit top-ups for BYOK and subscription activations for Premium. This is a single future addition, not needed until you monetize.
